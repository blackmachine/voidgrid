# VoidGrid Engine

**VoidGrid** — это высокопроизводительный тайловый (grid-based) движок рендеринга, написанный на Rust с использованием библиотеки Raylib. Он предназначен для создания интерфейсов, игр и визуализаций, имитирующих терминал, но с возможностями современного GPU-рендеринга: шейдерами, слоями, трансформациями и иерархией объектов.

---

## 🏗 Архитектура

Проект построен на принципе разделения данных (Model) и представления (View), объединенных общим фасадом.

### Основные компоненты

1.  **`VoidGrid` (Facade):**
    *   Главная точка входа (`lib.rs`).
    *   Владеет экземплярами `Grids` и `Renderer`.
    *   Скрывает сложность инициализации и синхронизации подсистем.

2.  **`Grids` (Data / State):**
    *   Отвечает за хранение состояния мира.
    *   Использует `SlotMap` для хранения ресурсов (Буферы, Атласы, Палитры, Шейдеры), что решает проблемы владения и циклических ссылок.
    *   Управляет иерархией объектов (Parent-Child) через систему `Attachment`.
    *   Не содержит кода, связанного с прямым рисованием на экран.

3.  **`Renderer` (View):**
    *   Отвечает за визуализацию данных из `Grids`.
    *   Управляет `RenderTexture` для off-screen рендеринга (для применения шейдеров к буферам).
    *   Реализует двухпроходный рендеринг и постобработку.

---

## 📂 Структура модулей

*   **`lib.rs`** — Публичный API библиотеки. Содержит структуру `VoidGrid`.
*   **`grids.rs`** — Логика управления состоянием (`Grids`). Методы манипуляции буферами, привязки детей, настройки шейдеров.
*   **`renderer.rs`** — Логика отрисовки. Содержит `Renderer`, управление `RenderTexture` и uniform-переменными шейдеров.
*   **`text_ops.rs`** — Трейт `TextOps`, расширяющий функционал `Grids` методами высокоуровневого вывода текста (с переносами, цветами, масками).
*   **`buffer.rs`** — Структура `Buffer` (хранит сетку `Character`) и `Attachment` (связи).
*   **`atlas.rs`** — Загрузка и конфигурация текстурных атласов (`Atlas`, `AtlasConfig`). Поддержка JSON-маппинга символов.
*   **`palette.rs`** — Система цветовых палитр (`Palette`). Поддержка именованных цветов и индексации.
*   **`shader.rs`** — Обёртка над GLSL шейдерами (`ShaderData`), управление Uniforms.
*   **`assets.rs`** — Загрузчики ресурсов с диска и система кэширования (`AssetCache`) для предотвращения дублирования загрузки.
*   **`input.rs`** — Обработка ввода (мышь, клавиатура, drag-n-drop) и декорации окна (`WindowChrome`).
*   **`types.rs`** — Базовые типы данных: `Transform`, `Blend`, `Mask`, `ColorRef`, ключи `SlotMap`.

## ⚙️ Ключевые механики

### 1. Буферы и Иерархия
Все данные хранятся в **Буферах**. Буфер — это 2D-сетка символов (`Character`).
*   **Иерархия:** Буферы могут быть прикреплены друг к другу (`attach`). Дочерние буферы наследуют координаты родителя и отрисовываются относительно него.
*   **Z-Index:** Поддерживается сортировка отрисовки как для независимых буферов, так и для детей внутри родителя.

### 2. Двухпроходный рендеринг (Shaders)
Движок поддерживает сложные эффекты (свечение, размытие, хроматическая аберрация) через шейдеры.
1.  **Pass 1 (Off-screen):** Буферы, к которым привязан шейдер, сначала рендерятся в отдельную текстуру (`RenderTexture`). При этом учитывается `padding` (отступ), чтобы эффекты могли выходить за границы символов.
2.  **Pass 2 (Screen):** Основное дерево сцены рисуется на экран. Вместо перерисовки содержимого "шейдерных" буферов, накладывается их заранее подготовленная текстура.

### 3. Атласы и Маппинг
Шрифты загружаются как текстурные атласы с JSON-конфигурацией.
*   Поддержка диапазонов (`"A".."Z"`), одиночных символов и строк.
*   **Варианты (Variants):** Один символ может иметь альтернативные начертания (например, `bold`, `inverted`), которые переключаются в коде.

### 4. Текстовые операции
Модуль `text_ops` предоставляет мощный API для вывода:
*   Автоматический перенос строк.
*   Направление письма (вправо, вниз, влево, вверх).
*   Авто-поворот символов в зависимости от направления письма.
*   Маскирование (текст внутри текста).

## 🚀 Предложения по оптимизации и рефакторингу

Этот раздел содержит анализ текущей кодовой базы и предложения по её улучшению.

### 🛠 Архитектурный рефакторинг

1.  **Event System (Система событий):**
    *   *Проблема:* Ввод (`input.rs`) жестко завязан на Raylib и опрашивается в `main.rs`.
    *   *Решение:* Создать очередь событий (`Event::Click`, `Event::Resize`, `Event::CharInput`) внутри `VoidGrid`. Это позволит отвязать логику приложения от конкретного бэкенда ввода и упростит реализацию UI-виджетов.

2.  **UI Framework:**
    *   *Проблема:* Сейчас кнопки и зоны перетаскивания реализуются вручную через проверку координат.
    *   *Решение:* Реализовать трейт `Widget`, который умеет рисовать себя в буфер и обрабатывать события. Создать базовые виджеты: `Button`, `Panel`, `Label`.

3.  **Абстракция Рендерера (Backend Agnostic):**
    *   *Проблема:* `Renderer` напрямую использует FFI вызовы Raylib.
    *   *Решение:* Выделить трейт `RenderBackend`. Это позволит в будущем заменить Raylib на `wgpu` или `macroquad` без переписывания логики `Grids`.

### ⚡ Оптимизация производительности

1.  **Batching (Пакетная отрисовка):**
    *   *Проблема:* Сейчас каждый символ рисуется отдельным вызовом `DrawTexturePro`. Для тысяч символов это создает огромный оверхед на вызовы CPU->GPU.
    *   *Решение:* Реализовать сборку вершинного буфера (Vertex Batching) для каждого атласа. Рисовать весь буфер (или даже слой буферов с одним атласом) одним вызовом отрисовки (`DrawMesh` или инстансинг). Это даст прирост FPS в 10-100 раз на больших объемах текста.

2.  **Dirty Flags (Флаги изменений):**
    *   *Проблема:* `Renderer` каждый кадр пересобирает сцену, даже если текст не менялся.
    *   *Решение:* Добавить флаг `is_dirty` в `Buffer`. Если буфер не менялся, можно кэшировать его отрисовку в `RenderTexture` (как сейчас делается для шейдеров, но для обычных буферов тоже).

3.  **Culling (Отсечение):**
    *   *Проблема:* Отрисовываются все символы буфера, даже если они за пределами экрана.
    *   *Решение:* В `draw_single_buffer` вычислять видимый диапазон индексов `x` и `y` на основе позиции камеры/окна и рисовать только видимые тайлы.

### 🧹 Чистота кода

1.  **Error Handling:**
    *   В `main.rs` много `unwrap()` и `expect()`. Стоит внедрить крейт `anyhow` или `thiserror` для более гранулярной обработки ошибок загрузки ресурсов.
2.  **Конфигурация:**
    *   Параметры окна и пути к ассетам зашиты в код. Имеет смысл вынести настройки движка в `Config` структуру или `.toml` файл.
