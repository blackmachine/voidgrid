# Void Transfer Protocol (VTP) Specification
**Version:** 1.0 (Draft)

## 1. Общие положения
VTP — это компактный бинарный протокол для удаленного управления тайловым терминальным движком VoidGrid. 
Протокол основан на концепции конечного автомата (State Machine). Клиент отправляет команды, изменяющие текущее состояние терминала (активный буфер, курсор, цвет) или вызывающие отрисовку.

### 1.1. Типы данных и Endianness
Все многобайтовые целые числа в протоколе передаются в формате **Little-Endian (LE)**.
* `u8`: 1 байт (беззнаковое)
* `u16`: 2 байта (беззнаковое, LE)
* `u32`: 4 байта (беззнаковое, LE)
* `color`: 4 байта (R, G, B, A в виде `u8`)
* `string`: строка, состоящая из `len: u16` и последующих `len` байт в кодировке UTF-8.

### 1.2. Состояние терминала (Terminal State)
Парсер на стороне движка обязан поддерживать следующий контекст:
* `active_buffer`: ID или ссылка на текущий буфер для отрисовки (по умолчанию - root).
* `cursor_x` (u32), `cursor_y` (u32): Текущая позиция курсора в тайлах.
* `fg_color` (color): Текущий цвет текста (по умолчанию White).
* `bg_color` (color): Текущий цвет фона (по умолчанию Blank/Transparent).
* `variant_id` (u8): Текущий индекс варианта шрифта (по умолчанию 0).

## 2. Команды (Opcodes)

### Группа 1: Управление состоянием (State Control)

#### `0x01` SET_BUFFER
Устанавливает активный буфер для последующих операций вывода. Сбрасывает курсор в (0, 0).
* **0-й байт:** `0x01` (Opcode)
* **1-2 байты:** `len: u16` (Длина имени буфера)
* **3..N байты:** `name: [u8]` (Имя буфера в UTF-8, без null-терминатора)

#### `0x02` SET_CURSOR
Устанавливает абсолютную позицию курсора в активном буфере.
* **0-й байт:** `0x02` (Opcode)
* **1-2 байты:** `x: u16`
* **3-4 байты:** `y: u16`

#### `0x03` SET_FG_COLOR
Устанавливает цвет переднего плана.
* **0-й байт:** `0x03` (Opcode)
* **1-4 байты:** `r: u8`, `g: u8`, `b: u8`, `a: u8`

#### `0x04` SET_BG_COLOR
Устанавливает цвет фона.
* **0-й байт:** `0x04` (Opcode)
* **1-4 байты:** `r: u8`, `g: u8`, `b: u8`, `a: u8`

#### `0x05` SET_VARIANT
Устанавливает активный вариант глифа (например, 0 - обычный, 1 - инвертированный).
* **0-й байт:** `0x05` (Opcode)
* **1-й байт:** `variant: u8`

### Группа 2: Вывод данных (Drawing)

#### `0x10` PRINT_CHAR
Печатает один символ по текущим координатам курсора, используя текущие цвета и вариант. После печати `cursor_x` автоматически увеличивается на 1.
* **0-й байт:** `0x10` (Opcode)
* **1-4 байты:** `code: u32` (Unicode codepoint или внутренний ID глифа)

#### `0x11` PRINT_STRING
Печатает строку по текущим координатам. Автоматически обрабатывает переносы строк `\n`. Курсор смещается в конец напечатанной строки.
* **0-й байт:** `0x11` (Opcode)
* **1-2 байты:** `len: u16` (Длина строки в байтах)
* **3..N байты:** `text: [u8]` (Текст в UTF-8)

### Группа 3: Загрузка ресурсов (Resource Management)

#### `0x20` LOAD_SCENE
Загружает новую сцену (ресурс-пак `.vpk`) прямо из потока байт. Эта команда блокирует отрисовку до полной загрузки данных.
* **0-й байт:** `0x20` (Opcode)
* **1-4 байты:** `size: u32` (Размер архива в байтах)
* **5..N байты:** `archive_data: [u8]` (Сырые байты ZIP-архива)

## 3. Обработка ошибок и фрагментации
* Пакеты могут приходить частями (особенно по TCP или Serial). Парсер должен накапливать байты до тех пор, пока их не станет достаточно для полного выполнения команды.
* Неизвестные опкоды игнорируются (выбрасывается 1 байт, парсер пытается найти следующую валидную команду).
